# Коннекторная система креплений — дизайн

Статус: принято, в работе (2026-07-22).
Заменяет пары «грид-пады + точные точки» одной абстракцией — **коннектором**.

## Зачем

Проблемы старой системы (по коду, не по ощущениям):

1. `exact_point`/`local_position` в `StructuralMountPad` авторились, но нигде не
   читались: матчинг (`grid_surface_util.gd`) смотрит только `cell+face+tag`,
   регистр их не сериализует, у снапа нет sub-cell пути. «Крепление на ступицу»
   не работало не из-за настройки — кода не было.
2. Логика конкретного типа детали зашита if-ладдерами минимум в 4 местах:
   совместимость сокетов (`wheel_socket↔wheel_plug` литералом в
   `grid_surface_util.gd:570`), `match part_kind` в авторинге, спец-кейсы в
   превью (вплоть до строки `"stationary_drill"`).
3. У детали нет канонической позы: игрок всегда получает orientation index 0.
4. Авторинг покрывал только блок/колесо/подвеску, требовал ручного сдвига
   модели в угол и последовательности кнопок-тумблеров.

## Модель

### ConnectorDefinition (ресурс)

Один коннектор = точка крепления. Поля:

- `id: String` — стабильный идентификатор внутри детали.
- `local_position: Vector3` — точка в лок. метрах детали (канонический фрейм).
- `local_direction: Vector3` — направление «наружу» (нормаль поверхности).
- `local_up: Vector3` — референс ролла (для несимметричных).
- `tag: String` — тип: `structural`, `wheel_socket`, `wheel_plug`, свои.
- `symmetric: bool` — деталь симметрична вокруг оси коннектора: угол ролла
  не важен (колёса, роторы). Иначе ролл шагами 90° от `local_up`.
- `grid_cell: Vector3i` + `grid_face: Face` + `is_grid: bool` — заполняются у
  сеточных коннекторов; это мост в топологию (окупансия/связность остаются
  целочисленными).

### Правила совместимости — данные, не код

`ConnectorRuleTable` (ресурс, `resources/connectors/connector_rules.tres`):
список пар тегов, которые мейтятся. Базово:
`structural↔structural`, `wheel_socket↔wheel_plug`. Новый тип детали =
новая строка, ноль правок в матчинге/снапе/превью.

### Синтез: миграция без правки .tres

`ElementArchetype.effective_connectors()`:

1. если авторские `connectors` заданы — они;
2. иначе синтез из `effective_mount_pads()` (пад → сеточный коннектор в центре
   грани, тег из `socket_tag`, `exact_point` → точный `local_position`);
3. для `FULL_SURFACE` — сеточный коннектор на каждой внешней грани футпринта.

Все существующие архетипы работают сразу; старый код падов выпиливается после
паритета.

## Рантайм

Ключевой факт разведки: сетка 0.5 м глубоко прошита в **топологии**
(окупансия `Vector3i→id`, связность, снап-валидация) и **мелко** в позе — поза
элемента выводится в двух функциях `GridPoseUtil.element_local_transform` /
`collider_local_transform` (`grid_pose_util.gd:38-68`). Кернел элементов —
чистый GDScript, натив — только террейн.

Решение: **(origin_cell, orientation_index) остаются источником истины для
топологии.** Точность добавляется полем `SimulationElement.pose_offset:
Transform3D` (identity по умолчанию):

- компонуется в двух функциях GridPoseUtil → автоматически едет в коллайдеры,
  COM/инерцию, меши, выбор рейкастом (метаданные коллайдера), seat offset;
- сериализация: `to_dict`/`from_dict` пишут только не-identity;
- якорные хелперы, считающие от клеток напрямую, правятся отдельно (~5):
  `PistonProjectionUtil.port_anchor_assembly_local` / `piston_axis_…`,
  `RotorProjectionUtil.rotor_axis_…`, `HingePlacementUtil.bend_axis_…` /
  `pivot_…`, `WheelProjectionUtil.mount_pad_anchor_assembly_local` — якорь
  берётся из **точки коннектора**, а не из центра грани;
- ролл разрешён только вокруг оси коннектора (иначе жёстко сматченные грани
  геометрически разъезжаются при целой топологии);
- чего не трогаем: `GridTransform`/24 ориентации (pose_offset — отдельное
  поле, ролл не раунд-трипится через orientation_index), окупансию,
  `RuntimeConnectivity`, интегрити.

### Матчинг

`get_surface_descriptors` отдаёт коннекторные дескрипторы. Сеточные мейтятся
смежностью граней (как раньше). Точечные — совпадением точки и
анти-параллельностью направлений после выравнивания. Совместимость — из
rule table. Спец-валидация колеса (`wheel_placement_util`) заменяется
генерик-матчингом.

### Снап в билд-режиме

Если у держимой детали точечный коннектор: ищем совместимые коннекторы у цели
под лучом, предлагаем позу «точка в точку, направления встречные». Ролл — те же
клавиши поворота, шаг 90°; `symmetric` — угол игнорируется. Сеточный магнит
остаётся как частный случай (сеточные коннекторы).

## Авторинг («домохозяечный»)

Плагин редактора, док «Мастер деталей», четыре шага:

1. **Закинь модель** — футпринт и масса считаются из AABB сразу при
   назначении; модель к началу координат двигать не надо, бак перецентрирует.
2. **Поставь как увидит игрок** — SubViewport с камерой-референсом; поза
   модели в этом вью запекается как каноническая. Автор не думает про оси.
3. **Кликни точки крепления** — клик по поверхности меша → рейкаст →
   коннектор в точке клика с нормалью поверхности. Теги/симметричность — в
   инспекторе маркера.
4. **Испечь** — валидация + `.tres` в `resources/archetypes/authored/`.

Тип детали — не enum: поведения (колесо, подвеска, пистон, ротор, …) —
подключаемые сабресурсы, каждое приносит свои коннекторы и определение.
Превью — гизмо плагина, а не инжектированные дочерние ноды.

## Каноническая поза

Архетип хранит каноническую ориентацию из авторинг-позы;
`tool_controller` сидит стартовый `selected_orientation_index` от неё с учётом
направления взгляда игрока — призрак появляется так, как деталь стояла в
авторинг-сцене, лицом к игроку.

## Порядок работ

1. Модель данных: ConnectorDefinition, rule table, синтез (паритет тестами).
2. Рантайм: матчинг → rule table; pose_offset; якоря из точек коннекторов;
   снап точечных коннекторов; выпил wheel-спец-кейсов.
3. Визард-плагин.
4. Каноническая поза → призрак.

Проверка каждой фазы: headless-тесты + запуск игры (глаз проверяет, тесты
охраняют).
